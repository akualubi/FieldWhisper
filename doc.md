
# AgriSentinel · 大田作物风险智能预警平台

## 一、背景描述

### 1.1 中国大田作物面临的系统性挑战

中国是世界第一大粮食生产国，玉米、小麦、水稻、大豆四大作物种植面积超过  **18 亿亩** ，构成国家粮食安全的根本盘。但这套庞大体系正在承压：

**第一，气候不确定性变成常态。** 2024 年黄淮海"烂场雨"导致小麦穗发芽和玉米霉变，2025 年东北强对流大风让数百万亩玉米一夜倒伏，2025 年夏华北卡脖旱让玉米抽雄期撞上 40℃ 极端高温。这些不再是"百年一遇"的孤立事件，而是逐年加密的新常态。

**第二，跨境与跨区病虫害北扩。** 草地贪夜蛾自 2019 年入侵后已在华北稳定越冬，南方锈病随台风路径年年北上，小麦条锈病、稻飞虱、大豆食心虫在新区域反复爆发。**传统的"经验防治"已经追不上病虫害的迁移速度。**

**第三，规模化经营与服务能力错配。** 全国土地流转率已超 35%， **百亩以上规模经营主体超过 600 万家** ——合作社、家庭农场、农垦农场、农业产业园成为种植主力。但县级农技推广体系人均服务面积仍超 5 万亩，老农技员退休、年轻人不下地， **专业知识供给严重短缺** 。

**第四，B 端机构亟需"可量化、可追溯、可决策"的农情数据。**

* **保险公司**承保面积越来越大，但灾前预警空白、灾后定损靠人走，赔付率失控
* **粮食收储企业**需要提前预判产量与品质，但数据来源散乱
* **政府农业部门**需要实时掌握辖区作物长势与灾情，但信息上报链条长、失真严重
* **农资经销商**需要预判区域用药需求，但库存调配靠经验

### 1.2 我们看到的市场机会

整个大田农业产业链里， **真正有付费能力、有数据需求、有规模化使用场景的，是 B 端机构** ——而不是分散的农户。

参考 YC 2026 RFS 中的 **AI-Native Service Companies** 路径，以及 **AI for Low-Pesticide Agriculture** 方向：B 端客户要的不是 App，而是**"嵌入他们业务流程的 AI 服务结果"**——保险公司要的是承保地块的实时风险报告，收储企业要的是产区品质预判，政府要的是辖区灾情一张图。

**我们的判断是：把"农情风险智能体"做成 B 端的 API 和服务接口，而不是 C 端的 App。**

### 1.3 重大风险 vs 日常灾害——两层风险，一套底座

无论服务哪类 B 端客户，底层风险结构是一致的：

**重大风险（低频高损失）**

* 极端天气：高温热害、强对流、持续干旱、烂场雨、寒潮霜冻
* 区域性病虫害爆发：草地贪夜蛾、南方锈病、小麦条锈病、稻飞虱大迁飞
* 关键生育期叠加灾害

**日常灾害（高频累积损失）**

* 常规病害：大斑病、白粉病、稻瘟病、纹枯病等
* 常规虫害：钻心虫、蚜虫、二化螟、食心虫等
* 长势异常：营养失衡、水分胁迫、除草剂药害

 **两类风险共用同一套智能体底座** ，但对不同 B 端客户输出不同的"风险产品"。

## 二、解决方案

### 2.1 一句话定义

**AgriSentinel 是面向农业 B 端机构的作物风险智能预警平台。我们用一套多 Agent 系统，融合气象、遥感、田间影像与历史灾害知识，为保险公司、合作社、农垦集团、政府农业部门和粮食企业，提供地块级、可追溯、可决策的大田作物风险情报服务。**

### 2.2 产品形态：不是 App，是"风险情报基础设施"

我们提供三种交付形态，覆盖不同类型的 B 端客户：

 **① SaaS 控制台** （适合合作社、农垦集团、产业园）

* Web 端管理后台，可视化呈现辖区所有地块的风险地图
* 风险等级、推荐措施、历史档案、效果追踪一站式

 **② API / 数据服务** （适合保险公司、粮食企业、农资分销商）

* 标准化 REST API：输入地块经纬度 + 作物类型 → 输出风险评分、灾害预测、建议措施
* 按调用量或按承保面积订阅计费

 **③ 政企定制部署** （适合县/市级农业农村局）

* 私有化部署，整合辖区气象、遥感、田间网格数据
* 输出"全县农情一张图"+ 灾情预警简报 + 决策辅助报告

### 2.3 核心 Agent 工作流

整个系统是 **多 Agent 协作的"感知-分析-决策-推送-进化"闭环** ，但每一层都根据 B 端场景做了重新设计：

**① 感知层 Agent**

* **气象 Agent** ：地块级气象数据融合（公共气象 API + 商业气象数据 + 自建气象站）
* **遥感 Agent** ：调用公开卫星遥感数据（Sentinel-2、Landsat），监测大范围长势 NDVI
* **田间影像 Agent** ：通过合作社派人/无人机/固定摄像头采集网格点照片，VLM 识别
* **作物档案 Agent** ：维护每个地块的作物品种、播种期、生育期、用药历史、产量曲线

**② 分析层 Agent**

* **重大风险监测 Agent** ：基于气象预测 + 生育期敏感性 + 历史共性灾害模式，对高温、大风、暴雨等做 24-72 小时提前预警
* **病虫害诊断 Agent** ：多作物 VLM 模型矩阵（玉米、小麦、水稻、大豆各一套），支持图像识别 + 历史样本匹配
* **区域共性分析 Agent** ：跨地块横向比较，发现"同一区域同时发病"的模式，预测扩散路径
* **置信度评估 Agent** ：每条结论附置信度评分，低置信度触发人工农艺师复核

**③ 决策层 Agent**

* 调用农艺知识库（农业农村部规程 + 各省农科院方案 + 主流农药数据库）
* 输出 **结构化结论** （不是聊天文本），供下游 B 端系统调用：
  ```
  {  "地块ID": "BJ-CP-001",  "作物": "玉米",  "风险等级": "高",  "风险类型": "高温热害",  "建议措施": ["清晨喷灌降温", "叶面喷磷酸二氢钾"],  "最佳行动窗口": "未来 48 小时",  "置信度": 0.87,  "数据来源": ["气象API-2026/05/25", "历史样本-2022皖北案例"]}
  ```

**④ 推送层 Agent（按 B 端客户定制）**

| 客户类型    | 推送形态                         |
| ----------- | -------------------------------- |
| 保险公司    | API 推送至承保系统，触发理赔预审 |
| 合作社/农垦 | 控制台风险地图 + 短信通知社员    |
| 政府部门    | 灾情简报 PDF + 大屏可视化        |
| 粮食企业    | 产区品质预判月报                 |
| 农资分销    | 区域用药需求预测，指导库存调配   |

**⑤ 进化层 Agent**

* 每次预警进入闭环跟踪：B 端客户反馈、实际灾情数据、产量结果
* 不准确的判断进入"错题本"，反向优化模型与知识库
* 多客户数据汇聚形成  **"全国大田作物风险知识图谱"** ——这是数据壁垒，也是 B 端续费的核心理由

### 2.4 多作物如何支持

我们不为每种作物从零做一套系统，而是 **模型矩阵 + 知识库分层** ：

* **底座层** （共用）：气象 Agent、遥感 Agent、Agent 编排框架、推送系统、向量检索
* **作物层** （独立）：每种作物一套
* 病虫害识别模型（VLM 微调或 prompt 工程）
* 生育期模型（积温-生育期对应表）
* 农艺知识库（防治方案、用药规范）
* 历史灾害样本库

 **黑客松 Demo 阶段** ：先做 **玉米 + 小麦** 两个作物的完整链路（这是华北种植户最关心的两个），其他作物作为"路线图未来 3 个月扩展"展示。

### 2.5 黑客松 Demo（一天能跑通的最小闭环）

**Demo 场景 ①：高温热害预警（重大风险，玉米）**

* 输入：一个虚拟地块经纬度 + 作物"玉米抽雄期"
* AI 链：气象 API → 检测连续 35℃+ → 匹配历史案例 → 生成预警
* 输出：B 端控制台风险地图变红 + API 推送 JSON + 短信文案

**Demo 场景 ②：小麦条锈病早期识别（日常灾害，小麦）**

* 输入：上传一张小麦叶片照片
* AI 链：VLM 识别 → 区分条锈病/叶锈病/缺素 → 检索相似历史样本 → 生成方案
* 输出：诊断报告 + 防治窗口 + 预计区域扩散趋势图

**Demo 场景 ③：保险公司视角的 API 调用**

* 输入：模拟保险公司请求"承保区域 1000 个地块的实时风险评分"
* AI 链：批量调用 → 返回结构化 JSON
* 输出：用 Postman/前端模拟接收方，展示**B 端集成的真实样子**

> **这第三个 Demo 极其重要** ——它是把"我们 to B"这句话**视觉化、可信化**的关键。评委里如果有产业方背景的，看到这个会立刻 get 到你们不是在做玩具。

### 2.6 商业化路径（重新设计的 B 端版本）

**核心收入模型：分层订阅 + 按量计费**

| 客户类型                    | 收费模式                | 单客户年收入估算           |
| --------------------------- | ----------------------- | -------------------------- |
| **农险公司**          | 按承保面积，¥1-2/亩/年 | 一家省级农险公司 ≈ 500 万 |
| **省级农垦集团**      | 私有化部署 + 年服务费   | 200-500 万/家              |
| **大型合作社/产业园** | SaaS 订阅 ¥3-5 万/年   | 3-5 万/家                  |
| **县农业农村局**      | 政府采购，年度框架协议  | 50-150 万/家               |
| **粮食收储企业**      | 数据订阅 ¥10-30 万/年  | 10-30 万/家                |
| **农资分销集团**      | 区域用药预测 API        | 20-50 万/家                |

 **市场规模快速测算** （路演用，简单算大账）：

* 全国农险保费规模 1500 亿+，AI 风控渗透率 1% = **15 亿市场**
* 全国 2000+ 个产粮县 × 政府采购均价 100 万 = **20 亿市场**
* 全国规模化合作社 220 万家，付费转化 1% × 3 万/家 = **6.6 亿市场**
* 合计可触达市场规模 **40 亿+**

 **GTM 优先级** ：

1. **首推保险公司** ——付费意愿最强、合同金额最大、效果最容易量化（赔付率下降）
2. **次推地方政府** ——决策周期长但金额稳定，且自带"数字农业"政策红利
3. **再扩合作社/农垦** ——客单价低但可规模化复制

### 2.7 差异化（为什么 B 端会买我们）

| 维度       | 传统农业信息化厂商 | 通用 AI 服务（如 GPT API） | AgriSentinel                         |
| ---------- | ------------------ | -------------------------- | ------------------------------------ |
| 数据资产   | 静态行业数据库     | 无                         | **持续进化的区域灾害知识图谱** |
| 决策输出   | 数据展示，决策靠人 | 通用回答，无农艺深度       | **结构化决策建议 + 置信度**    |
| 多作物支持 | 单作物专精         | 通用但浅                   | **模型矩阵 + 共享底座**        |
| B 端集成   | 私有协议、定制开发 | 通用 API                   | **标准化 API + 行业 schema**   |
| 闭环反馈   | 无                 | 无                         | **进化层 Agent 持续优化**      |

## 三、给后续实现的关键提示

**1. 多作物不要全做，挑 2 个深做。** 黑客松一天做不了 4 个作物，**玉米 + 小麦**最稳（华北、黄淮海、东北全覆盖）。其他作为"产品路线图"展示，评委不会追问。

**2. B 端 Demo 必须演"API 视角"。** 在 Demo 里专门设计一个"切换身份"的演示——切到"保险公司视角"，让评委看到 JSON 接口、批量调用、风险评分这些 B 端典型场景。**这是与 C 端项目拉开差距的关键一刀。**

**3. 商业模型用真实数字。** 农险保费 1500 亿、产粮县 2000+、土地流转率 35%——这些数字百度可查，但放在 BP 里立刻显得专业。**别用"市场很大"这种空话。**

**4. 数据壁垒要说清楚。** B 端评委会问"为什么大厂不能做"。答案是： **我们做的不是模型，是数据反馈闭环** ——用得越多越准，先入场就形成网络效应。这个论证逻辑一定要在 BP 里讲。

**5. 别忘了合规话题。** 涉及农业数据、保险数据、政府数据时，评委可能会问数据合规。提前准备一句话："所有数据脱敏存储，遵循《数据安全法》《个人信息保护法》，政府/保险客户支持私有化部署。"——回答有就行，不用展开。

**6. 团队分工对 B 端友好。** 8 个人里至少要有 1 人 **专门负责"B 端商务故事"** ——他不写代码，他在 15:00-17:00 拉评委时专门讲商业模式和客户场景。这个角色对 B 端项目极其关键。
